The problem

Agents don't ship at human pace.

Coding agents produce more change per hour than anyone can absorb. The bottleneck has moved: writing the code is cheap now; understanding it isn't. And whoever merges it still owns every line.

The problem

Alphabetical order is not an explanation.

Our review tools were built for human-sized diffs. They sort the change by file path and call it a day, so a lockfile and a payment-critical retry loop get equal billing.

You start at .github/ and hope the design reveals itself by test/.

The problem

The diff is complete. Completeness isn't comprehension.

A diff answers what changed, line by line. It never answers why, or what belongs together, or which of those 47 files deserves your next ten minutes.

So reviewers reverse-engineer intent from hunks: scrolling, guessing, rebuilding the design in their heads, for a branch nobody on the team actually wrote.

The tool

ndrstnd

A local comprehension workspace for agent-produced changes. It turns a branch diff into an evidence-linked story, and reads it to you the way you'd want a colleague to: claims first, meaning second, evidence on demand.

$ndrstnd review codex/checkout-retry-hardening --repo .
  • Runs entirely on your machine
  • Open source · Apache-2.0
  • Self-contained artifacts, private to your working copy

Before

A single capture attempt per intent. Any gateway timeout surfaced to the caller as a failed payment, and manual retries risked charging the customer twice.

After

Transient failures retry up to maxAttempts with jittered exponential backoff, and every attempt reuses the same idempotency key, so a retry can never double-charge.

The point

Comprehension, not critique.

ndrstnd deliberately doesn't leave review comments, doesn't score your agent, and never edits the branch. It explains the implementation story, the evidence, and the risk signals, then gets out of the way.

The judgment stays yours.

Under the hood

From branch to story, in five steps.

01

Diff

Compare the branch to its Git merge-base, or to --base, or the uncommitted worktree.

02

Classify

Parse and index every hunk; separate meaningful changes from lockfiles, generated, and vendored code.

03

Analyze

Ask Codex for the narrative: chapters, timeline, test plan. Every claim is bound to real hunks and validated against a schema.

04

Render

Bake models, syntax highlighting, and UI into one self-contained HTML artifact.

05

Open

Write it to the repo's .ndrstnd/ directory and open it in your browser. Old artifacts expire after seven days.

Node ≥ 22 · TypeScript · everything runs locally: the diff goes to Codex through your own session, and nothing else leaves the machine.

Under the hood

Four decisions that shaped it.

1
Artifact, not app

A review is a static HTML file, not a running server. Nothing to install for readers, nothing to keep alive, nothing that phones home.

2
Never fabricate

The analysis validates against a strict schema, with one repair turn. If it still fails, ndrstnd writes nothing and tells you. It never invents a story it can't back.

3
Evidence-linked by construction

Claims that can't point at real hunks don't ship. Every sentence in the story is a link to the code that proves it.

4
Zero token custody

ndrstnd reuses Codex's authenticated session and never stores a credential of its own. Auth stays someone else's well-solved problem.

Under the hood

Reviews that read from a phone.

Agents finish while you're away from your desk. So the artifact treats mobile as a first-class layout, not a shrunken desktop: chapters read like an article, the zoom dial docks as a floating pill at the bottom, and the page never scrolls sideways.

It pairs naturally with Codex Remote: kick off the work from anywhere, and read the finished story on your phone before you're back at a keyboard.

Appendix · Architecture

A small TypeScript CLI, end to end.

server/cli.ts

Command surface: auth, skill install, review with scope flags. Scope confirmation and inferred-base warnings live here.

server/git.ts · git-model.ts

Runs Git, finds the merge-base, parses hunks, and classifies files as meaningful vs. low-signal (lockfiles, generated, vendored, binary).

server/analysis-core.ts · analyze.ts

Builds the prompts, calls Codex, validates the narrative against the schema, and runs the single repair turn.

web/page.ts + models

Renders the artifact: evidence and test-plan presentation models, language heuristics, and the whole UI baked into one HTML file.

skill-assets/ndrstnd

The Codex skill: teaches the agent to choose a review scope, default to a conversation export, and hand back the artifact link.

test/

Unit and DOM tests over the models and rendering; the artifact is asserted against real generated pages.

Appendix · Never fabricate

The analysis is a contract, not a vibe.

  • 1
    Strict schema

    The narrative must parse and validate (chapters, timeline steps, test themes, evidence references) or it doesn't exist.

  • 2
    One repair turn

    Invalid output gets exactly one chance to fix itself, with the validation errors echoed back. No endless retry loops.

  • 3
    Evidence must resolve

    Every claim's hunk references and focus ranges are checked against the real diff. Dangling evidence invalidates the claim.

  • 4
    Fail loudly

    If validation still fails, no artifact is written. You get an explicit error instead of a confident fiction.

“If the analysis fails, it tells you so: it never fabricates a story it can't back with evidence.”

Appendix · Scopes & signal

Review exactly the change you mean.

Choosing the base

  • A

    ndrstnd review my-branch: the branch against its merge-base. The default; it reviews only what the branch introduced.

  • B

    --base main: the checked-out worktree against a branch, staged and unstaged changes included.

  • C

    --uncommitted: just what isn't committed yet (an alias for --base HEAD).

  • D

    --base empty: everything, for brand-new repositories. When a base is inferred, the CLI says so before analyzing.

Separating signal from noise

Files are classified before analysis: lockfiles, generated code, vendored dependencies, and binaries are low-signal. They stay collapsed in every view and out of the narrative's attention, but always one click away and always counted.

Omitted, never hidden: the same rule as the ⌁ divider, applied at file scale.

Appendix · Context

The decisions, not just the diff.

Conversation-informed analysis

Pass --conversation with the agent's session export and the implementation dialogue becomes part of the story: why an approach was chosen, what was tried and abandoned, what the human asked for along the way.

Answers carry provenance (based on the branch, the conversation, or both), so you always know what a statement rests on.

Analysis lenses

Re-run the analysis through a different lens when the default story isn't the story you need. Each run is an immutable revision; nothing gets overwritten.
